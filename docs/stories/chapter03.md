# 第3章：データ構造研究所

**〜 情報を整理する魔法 〜**

**カテゴリ：データ型・構造（Data Types & Structures）**
**登場用語：String, Boolean, Integer, Number, Null, Undefined, Array, Object, Key-value pair, Variable, Constant, Data type, Primitive, Tuple, Set, Map, Hash, Index, Schema, Record, Field, Enum, Interface, Type alias, Union type, Stack, Queue, Tree, Graph, Vector**

---

## シーン1：研究所への招待

```
図書館を出ると、白い建物が見えてきた。
「データ構造研究所」という看板がかかっている。

「ここでは、データをどうやって整理するか
研究しているんだ」とビットくんが言った。

扉を開けると、白衣を着た人物が出迎えた。

「ようこそ！私はストラクト博士。
データを美しく整理する方法を研究しています」

「データを整理……？」

「そう。データにも『型』があるんです。
文字なのか、数字なのか、
正しいか間違いか……
型を知れば、データを正しく扱えますよ」
```

---

## シーン2：基本の型たち

```
最初の部屋には、いくつかの棚があった。

「まず、Primitive（プリミティブ）——
基本データ型を紹介しましょう」

「String（文字列）は、文字の並びです。
"Hello" や "こんにちは" がこれですね。
必ずクォーテーションで囲みます」

「Number（数値）は、数字を扱います。
42 や 3.14 のような数ですね。
Integer は整数だけを指します」

「Boolean（ブーリアン）は、
true（真）か false（偽）の2つだけ。
『はい』か『いいえ』の答えに使います」

「シンプルだけど、これが基本なんだね」
```

**💡 用語登場：String, Number, Integer, Boolean, Primitive, Data type**

---

## シーン3：「ない」の表現

```
「興味深いのは、『ない』という状態も
データ型で表現することです」

博士が空っぽの箱を見せた。

「Null（ヌル）は、
『意図的に何もない』状態です。
箱は用意したけど、中身を空にしておいた、
というイメージですね」

「Undefined（アンディファインド）は、
『まだ定義されていない』状態。
そもそも箱を用意し忘れた、という感じです」

「どちらも『ない』だけど、
意味が違うんだ……」

「この違いを理解すると、
エラーの原因を見つけやすくなりますよ」
```

**💡 用語登場：Null, Undefined**

---

## シーン4：データの入れ物

```
次の部屋には、箱とラベルがたくさんあった。

「Variable（変数）は、
データを入れておく箱です。
中身は後から変えられます」

let score = 0;   // 最初は0点
score = 100;     // 100点に変更！

「Constant（定数）は、
一度入れたら変えられない箱。
ずっと同じであるべき値に使います」

const PI = 3.14159;  // 円周率は変わらない

「変わるかもしれないものは変数、
絶対変わらないものは定数。
使い分けが大切なんですね」
```

**💡 用語登場：Variable, Constant**

---

## シーン5：配列という引き出し

```
長い棚に、番号のついた引き出しが並んでいた。

「Array（配列）は、
同じ種類のデータを順番に並べた構造です」

["りんご", "みかん", "ぶどう"]

「引き出しの番号を Index（インデックス）といいます。
プログラミングでは0から数えるのが特徴です」

インデックス: 0      1       2
データ:     りんご  みかん   ぶどう

「0から数えるの？」

「はい、これはよくある勘違いポイントです。
最初の要素は1番目ではなく、0番目なんです」
```

**💡 用語登場：Array, Index**

---

## シーン6：オブジェクトという整理棚

```
別の部屋には、ラベル付きの棚があった。

「Object（オブジェクト）は、
関連するデータをまとめた構造です。
Key-value pair（キーと値のペア）で
整理するんですよ」

const person = {
  name: "ビットくん",    // nameがKey、"ビットくん"がValue
  age: 1,
  job: "ガイド"
};

「名前、年齢、職業……
ひとつの『人』に関する情報を
まとめて管理できるんだね」

「これが Record（レコード）の考え方です。
ひとつひとつの項目を Field（フィールド）と
呼びます」
```

**💡 用語登場：Object, Key-value pair, Record, Field**

---

## シーン7：特別なコレクション

```
「もっと便利なデータ構造もありますよ」

博士が図を描きながら説明した。

「Set（セット）は、重複のないデータの集まり。
同じものを2回入れても、1個しか残りません」

「Map（マップ）は、
キーと値のペアを効率よく管理します。
Hash（ハッシュ）とも呼ばれます」

「Tuple（タプル）は、
固定長の配列で、順番に意味があります。
座標 (x, y) のようなものですね」

「それぞれ得意なことが違うんだ……」
```

**💡 用語登場：Set, Map, Hash, Tuple**

---

## シーン8：型の設計図

```
「大きなプロジェクトでは、
データの『設計図』を作ります」

博士がホワイトボードに書いた。

「Schema（スキーマ）は、
データ構造の設計図です。
どんな項目があって、それぞれ何型か
定義しておくんです」

「Interface（インターフェース）は、
オブジェクトの形を定義します。
『この形に従ってね』という約束事です」

interface Person {
  name: string;
  age: number;
}

「Type alias（型エイリアス）は、
型に別名をつけること。
Union type（ユニオン型）は、
複数の型のどれかを許容します」

type Status = "active" | "inactive" | "pending";
```

**💡 用語登場：Schema, Interface, Type alias, Union type**

---

## シーン9：列挙と選択肢

```
「Enum（イーナム・列挙型）は、
決まった選択肢だけを許す型です」

enum Color {
  Red,
  Green,
  Blue
}

「信号機の色は、赤・青・黄しかないですよね。
それ以外の値が入らないよう、
制限をかけることができます」

「間違ったデータを防げるんだね」

「その通り！
型システムは、
間違いを未然に防ぐ安全装置なんです」
```

**💡 用語登場：Enum**

---

## シーン10：高度なデータ構造

```
研究所の奥には、立体的な模型があった。

「ここからは上級者向けですが、
紹介しておきましょう」

「Stack（スタック）は、
本を積み上げるイメージ。
最後に入れたものを、最初に取り出します」

「Queue（キュー）は、
レジの行列のイメージ。
最初に並んだ人から順番に対応します」

「Tree（ツリー）は、
家系図のような階層構造です」

「Graph（グラフ）は、
駅の路線図のような、つながりの構造」

「Vector（ベクトル）は、
方向と大きさを持つデータ。
ゲームでキャラクターを動かすときに使います」

「データ構造って、こんなに種類があるんだ……！」
```

**💡 用語登場：Stack, Queue, Tree, Graph, Vector**
